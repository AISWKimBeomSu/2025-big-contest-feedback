🚨 발견된 핵심 이슈 (심각도 순)

#1 LLM 이중 호출 구조Critical
LangGraph Agent가 Gemini로 도구를 선택한 뒤, tools.py 내부에서 call_gemini_llm()으로 다시 Gemini를 직접 호출합니다. 한 질문에 LLM이 2~3회 호출되어 비용 2~3배, 응답시간 10초 이상 소요됩니다.
#2 LLM 모델 버전 불일치Critical
streamlit_app.py의 Agent는 gemini-2.5-flash를, tools.py의 직접 호출은 gemini-2.0-flash-exp를 사용합니다. 같은 파이프라인에서 서로 다른 모델이 동작하여 응답 톤/품질 불일치가 발생합니다.
#3 데이터 로딩 단일 장애점High
9개 CSV가 Google Drive ZIP 1개에 전적으로 의존. Drive 다운로드 제한, 네트워크 오류, URL 만료 시 앱 전체가 작동 불능. 폴백 메커니즘 없음.
#4 -999999.9 → NaN 변환 불완전High
df_all_join에만 .replace(-999999.9, np.nan) 수행. 유동인구 데이터프레임에는 적용되지 않아 -999999.9가 그대로 분석에 사용될 수 있음.
#5 store_id 프롬프트 인젝션 취약High
사용자 입력 store_id가 검증 없이 f-string으로 LLM 프롬프트에 삽입. 정규식 필터 등 기본 입력 검증 부재.
#6 ToolExecutor 이중 래핑Medium
tools.py의 @tool 함수를 ToolExecutor에서 다시 @tool wrapper로 감쌈. 2중 래핑으로 디버깅 난이도 상승.
#7 대화 히스토리 무한 증가Medium
전체 히스토리를 agent.invoke()에 전달. 분석 보고서가 수천 토큰이므로 3~4회 대화만으로 토큰 한도 초과 가능.
#8 매직넘버 하드코딩Medium
재방문율 30%, threshold max-5.0, apply_emphasis의 0.7승 등 통계적 근거 없는 상수 산재.

🔍 카테고리별 상세 분석

1. 아키텍처 설계 (45~65점)

LangGraph ReAct 에이전트를 중심으로 5개 도구를 라우팅하는 설계 방향성은 올바릅니다. 그러나 핵심 문제는 "도구가 분석만 하고 결과를 반환"해야 하는데, 도구 내부에서 또 다른 LLM을 호출하여 보고서까지 작성한다는 점입니다. Agent의 역할(도구 결과를 종합하여 최종 응답 생성)과 충돌하여, Agent가 도구의 LLM 출력을 다시 한번 가공하는 3중 구조가 됩니다.

개선 방향: 도구는 순수 데이터 분석만 수행 → 구조화된 JSON/dict 반환 → Agent(또는 별도 LLM 1회)가 최종 보고서 생성. LLM 호출이 1~2회로 감소합니다.

2. 데이터 엔지니어링 (55점)

데이터 JOIN 무결성은 우수 — 3개 데이터셋(4,185 가맹점 × 86,590 월별 레코드) 간 ENCODED_MCT 기준 100% 매칭 확인. @st.cache_data 적용으로 반복 로딩 방지.

그러나 -999999.9 처리가 df_all_join에만 적용되어, 유동인구 데이터에서 이 값이 실제 분석에 사용될 수 있습니다. ZIP 다운로드 실패 시 10개의 None을 반환하는 방식은 안티패턴입니다.

3. 분석 로직의 통계적 신뢰도 (60점)

백분위 기반 비교 분석 자체는 합리적이나, apply_emphasis에서 사용하는 0.7승 변환의 통계적 근거가 없습니다. 왜 0.7인지(0.5나 0.8이 아닌 이유) 설명이 없습니다. 재방문율 임계값 30%, 고객 세그먼트 threshold max_value - 5.0 등도 도메인 전문가 검증이 필요합니다.

4. 보안 (40점)

API 키 Streamlit Secrets 관리는 적절하나, store_id가 검증 없이 DataFrame 필터링과 LLM 프롬프트 f-string에 직접 삽입됩니다. 정규식 패턴 매칭(예: ^[A-Z0-9]{10}$) 같은 기본 입력 검증이 필요합니다.

5. 확장성 & 유지보수성 (45~50점)

새 분석 도구 추가 시: (1) tools.py에 함수 작성 → (2) ToolExecutor에 메서드 추가 → (3) get_all_tools()에 wrapper 추가 → (4) system_prompt에 가이드 추가. 4곳을 동시에 수정해야 하므로 도구 자동 등록(registry) 패턴 도입이 필요합니다.

6. 코드 품질 세부사항

PERSONA_MAP: 10개 페르소나 정의는 도메인 이해도가 높음. 그러나 코드에 하드코딩되어 있어, 업종 추가/변경 시 코드 수정이 필요합니다. 외부 JSON/YAML 설정 파일로 분리가 바람직합니다.

프롬프트 관리: system_prompt와 각 도구별 LLM 프롬프트가 코드 안에 수백 줄의 문자열로 존재합니다. 프롬프트 템플릿 파일 분리, 버전 관리, A/B 테스트 기반 최적화 체계가 필요합니다.

중복 코드: _get_store_basic_info()는 잘 추출되었으나, 데이터 포맷팅(fmt(), normalize_columns())이 도구마다 중복 정의되어 있습니다.

📋 종합 평가

🟢 잘한 점

소상공인 마케팅이라는 실용적 도메인에서 5가지 분석 관점을 체계적으로 분해한 것은 우수합니다. LangGraph + 도구 라우팅이라는 에이전트 아키텍처 방향성도 올바르고, Streamlit을 활용한 빠른 프로토타이핑과 배포도 효과적입니다. 데이터 3종 간의 JOIN 무결성(100%)도 좋습니다.

🔴 개선 필수

LLM 이중 호출 구조는 비용·성능·일관성 모두에서 문제입니다. 도구는 데이터 분석만, LLM은 보고서 생성만 담당하도록 역할 분리가 필요합니다. 입력 검증, NaN 처리 완전성, 테스트 코드 부재는 프로덕션을 위해 반드시 해결해야 합니다.
